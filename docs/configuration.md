# TermX Web (`web.env`) Configuration Reference

Every runtime-configurable parameter of the TermX web frontend, in one place. Use it to fill in the
`web.env` file of a deployment.

Sources: [`app/src/environments/env.js`](../app/src/environments/env.js) (the `${PLACEHOLDER}` template),
[`app/src/environments/environment.prod.ts`](../app/src/environments/environment.prod.ts) (defaults and
mapping) and [`app/src/environments/environment.base.ts`](../app/src/environments/environment.base.ts)
(the typed schema). The public [installation guide](https://tutorial.termx.org/en/installation-guide)
("web.env" tab) is a companion, but **this file (read from the code) is authoritative** тАФ the tutorial
predates the Kodality тЖТ TermX rename and shows stale `kodality.dev` URLs and a legacy `"dummy"` auth
convention. The server-side counterpart is
[`termx-server/docs/features/deployment-configuration.md`](https://github.com/termx-health/termx-server/blob/main/docs/features/deployment-configuration.md).

---

## How configuration is applied

termx-web is configured at **runtime, not build time** тАФ the same image runs in any environment. On
container start the entrypoint runs `envsubst` over `assets/env.js`, substituting every `${VAR}` from
the environment, producing `window.twConfig`. `environment.prod.ts` reads `twConfig` and applies a
built-in default wherever a value is empty/unset.

```
web.env (env vars)  тФАтФАenvsubstтФАтФАтЦ╢  assets/env.js  тФАтФАтЦ╢  window.twConfig  тФАтФАтЦ╢  environment.prod.ts
```

- **Plain-string vars** (`TERMX_API`, `BASE_HREF`, `OAUTH_ISSUER`, тАж) are used verbatim.
- **JSON-typed vars** (`EMBEDDED`, `GUEST_DISABLED`, `UI_LANGUAGES`, `CONTENT_LANGUAGES`,
  `EXTRA_LANGUAGES`, `I18N_OVERLAYS`) are declared as `json:${VAR}` and **must be valid JSON**. An empty or malformed
  value is dropped and the built-in default applies.
- **Changes take effect on container restart** тАФ `env.js` is regenerated by the entrypoint and served
  `no-store`. No image rebuild is needed.

## 1. Backend connection (required)

| Env var | Default | Purpose |
|---|---|---|
| `TERMX_API` | `/api` | Base URL of the TermX server API. Same-origin `/api` when web and server share a host; otherwise the absolute API URL. |
| `BASE_HREF` | `/` | App base path. Set to `/termx/` (note trailing slash) when the app is served on a sub-path, e.g. `https://site.org/termx/`. |

## 2. Authentication тАФ OIDC (required in production)

| Env var | Default | Purpose |
|---|---|---|
| `OAUTH_ISSUER` | *(required)* | OpenID Connect issuer / realm URL used as the OIDC authority (e.g. a Keycloak realm). |
| `OAUTH_CLIENT_ID` | *(required)* | Public OIDC client id registered in that realm (e.g. `term-client`). |
| `OAUTH_SCOPE` | `openid profile offline_access` | Scopes requested at login. |

## 3. Access control

| Env var | Type | Default | Purpose |
|---|---|---|---|
| `GUEST_DISABLED` | JSON bool | `false` | When `false`, unauthenticated visitors are loaded as a read-only **guest** (the server returns guest userinfo). Set `true` to require login for any access. |
| `EMBEDDED` | JSON bool | `false` | Embedded mode for hosting the UI inside another app: no guest login prompt and links open in the same tab. |

> Guest access is governed by `GUEST_DISABLED` plus the server's guest support тАФ there is no special
> `OAUTH_ISSUER="dummy"` mode in the current frontend (that is a legacy convention from the old tutorial).

## 4. Languages / i18n

| Env var | Type | Default | Purpose |
|---|---|---|---|
| `DEFAULT_LANGUAGE` | string | `en` | UI language used when the browser locale is not in `UI_LANGUAGES`. Also the Angular `LOCALE_ID`. |
| `ENGLISH_LOCALE` | string | `en-GB` | Regional locale used to **format dates** for the English UI language тАФ `en-GB` тЖТ `15 Jun 2026` / `15/06/2026` (day-first), `en-US` тЖТ `Jun 15, 2026` / `6/15/2026` (month-first). Does not change translations, only date order. |
| `UI_LANGUAGES` | JSON array | `["en","et","lt","de","fr","nl","cs"]` | Selectable interface languages. |
| `CONTENT_LANGUAGES` | JSON array | = `UI_LANGUAGES` | Languages offered in multilingual input fields. |
| `EXTRA_LANGUAGES` | JSON object | `{}` | Custom content languages with their label per UI language, e.g. `{"pl":{"en":"Polish","fr":"Polonais"}}`. |
| `I18N_OVERLAYS` | JSON array | `[]` | Extra translation catalogues merged **on top of** the bundled `assets/i18n/{lang}.json`, e.g. `["/assets/locales/{lang}.json"]`. `{lang}` is substituted per language; entries load in order and later ones win. Merging is deep, so overriding one key leaves its siblings intact, and a source that 404s is skipped тАФ a catalogue need not cover every language. Lets a deployment reword or add strings without editing the bundled files. |

## 5. External tools (optional)

| Env var | Default | Purpose |
|---|---|---|
| `SWAGGER_URL` | `/swagger/` | Swagger UI endpoint (API docs link). |
| `CHEF_URL` | `/chef` | FSH Chef service endpoint. |
| `CHEF_FHIR_VERSION` | `5.0.0` | FHIR version passed to Chef. |
| `PLANT_UML_URL` | `/plantuml` | PlantUML server endpoint (diagram rendering). |
| `FML_EDITOR` | `/fml-editor` | FHIR Mapping Language editor endpoint. |
| `FHIR_UML_CONVERTER_API` | *(empty)* | Base URL of the external [fhir2uml](https://fhir-uml-converter.online) converter. When set, the StructureDefinition **UML** diagram tab appears (it renders a class diagram from the version's FHIR content via this service); empty/unset hides the tab. |

## 6. SNOMED CT browsing links (optional)

External URLs the UI links to for SNOMED content. Empty by default (the related links are hidden).

| Env var | Default | Purpose |
|---|---|---|
| `SNOWSTORM_URL` | *(empty)* | Snowstorm terminology server the UI browses. |
| `SNOWSTORM_DAILY_BUILD_URL` | *(empty)* | Daily-build Snowstorm server. |
| `SNOMED_BROWSER_URL` | *(empty)* | SNOMED browser UI URL. |
| `SNOMED_BROWSER_DAILY_BUILD_URL` | *(empty)* | Daily-build SNOMED browser UI URL. |

## 7. Minimal production `web.env`

```bash
# required
TERMX_API=https://your-host/api
BASE_HREF=/
OAUTH_ISSUER=https://auth.your-host/realms/terminology
OAUTH_CLIENT_ID=term-client

# common optional
DEFAULT_LANGUAGE=en
ENGLISH_LOCALE=en-GB
UI_LANGUAGES=["en","et"]
GUEST_DISABLED=true
```

When web and server are served from the same host behind one reverse proxy, the path defaults
(`TERMX_API=/api`, `SWAGGER_URL=/swagger/`, `CHEF_URL=/chef`, `PLANT_UML_URL=/plantuml`,
`FML_EDITOR=/fml-editor`) usually need no change тАФ just point the proxy at the matching upstreams
(see the [server deployment reference](https://github.com/termx-health/termx-server/blob/main/docs/features/deployment-configuration.md#6-reverse-proxy-paths-reference)).

## 8. Gotchas

- **JSON vars must be valid JSON, unquoted-key-free.** `UI_LANGUAGES=["en","et"]`,
  `GUEST_DISABLED=true`, `EXTRA_LANGUAGES={"pl":{"en":"Polish"}}`. A blank or malformed value is
  silently dropped and the built-in default applies тАФ so a typo in `UI_LANGUAGES` quietly reverts to
  the full default list rather than erroring.
- **`BASE_HREF` needs the trailing slash** (`/termx/`) when serving on a sub-path, or asset/base-href
  resolution breaks.
- **`OAUTH_ISSUER` / `OAUTH_CLIENT_ID` have no defaults.** Omitting them in production yields a
  non-functional login. They must match a real OIDC client.
- **Restart, don't rebuild.** Editing `web.env` and restarting the container is enough; `env.js` is
  re-substituted on start and served `no-store`.
- **`ENGLISH_LOCALE` only affects the English language.** Other UI languages (et, ru, тАж) keep their
  own native date formats; this knob just chooses en-GB vs en-US for English.
- **Build-time-only flags are not `web.env` vars.** `appVersion`, `production` and `yupiEnabled`
  (dev-only permissive auth) come from the build/profile, not from `env.js`.
